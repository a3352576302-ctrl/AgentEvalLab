# AgentEvalLab 项目目标

## 1. 当前定位

AgentEvalLab 当前不是一个简单 Demo，而是一个面向 AI Agent 测试开发岗位的工程化评测原型。

它已经具备以下基础能力：

- 使用 YAML 管理 Agent 测试用例。
- 使用 pytest / run_report.py 批量执行测试。
- 支持 DeepSeek、MiniMax 等 OpenAI 兼容 API 的真实模型调用。
- 支持工具调用、参数检查、轮次检查、安全检查、Token 成本检查等 L1-L6 断言。
- 支持故障注入、失败归因、模型横向对比。
- 已接入 GitHub Actions，框架自身 197 条自动化测试全部通过。

当前项目最准确的描述是：

> 面向 AI Agent 应用上线前的自动化评测与回归测试平台原型。

它已经超过“能跑起来”的 Demo 阶段，但距离真实生产落地平台，还需要补齐真实业务场景、评测数据体系、生产稳定性、可视化报告、安全治理和团队协作能力。

## 2. AgentEvalLab 与 Demo 的区别

Demo 通常是“我做了一个功能，看起来能跑”。

AgentEvalLab 的目标是“我做了一套评测系统，可以反复、批量、可对比地测试 Agent 能力”。

| 维度 | 普通 Demo | AgentEvalLab |
| --- | --- | --- |
| 目标 | 展示一个功能能用 | 系统评测 Agent 在多场景下的表现 |
| 用例 | 通常手写几个样例 | YAML 管理多条测试用例 |
| 执行方式 | 手动点一下、跑一次 | pytest / run_report.py 批量执行 |
| 判断结果 | 人看输出觉得对不对 | L1-L6 自动断言 |
| 工具调用 | 可能只是简单模拟 | 检查工具序列、参数、轮次、安全和成本 |
| 稳定性 | 不一定能回归 | 自动化测试和 CI 保证框架稳定性 |
| 可对比性 | 很难横向比较 | 可对比不同模型的任务完成率、延迟、Token 成本 |
| 工程化 | 代码能跑即可 | 有报告、故障注入、失败归因和持续集成 |

一句话总结：

> Demo 证明“能跑”，AgentEvalLab 证明“能测、能复现、能对比、能定位问题”。

## 3. 距离真实落地还差什么

### 3.1 真实业务场景不足

当前用例主要是自定义 Agent 测试场景，还不够像真实公司里的业务任务。

真实落地需要补齐：

- 明确目标行业，例如客服 Agent、办公 Agent、代码 Agent、搜索 Agent、数据分析 Agent。
- 建立真实任务集，例如查订单并退款、根据知识库回答客户问题、生成 SQL 并解释结果。
- 覆盖真实用户的模糊表达、错误输入、越权请求、长上下文、多轮追问。
- 给每条用例标注业务优先级，例如核心链路、普通链路、低频异常链路。

后续目标：

> 把现有用例扩展成客服、搜索、文件、安全、数据分析、多工具规划等多个场景集。

### 3.2 评测数据集还不够标准化

当前 YAML 用例已经具备测试样本雏形，但还不算成熟 benchmark。

真实落地需要：

- 用例版本管理，例如 dataset_v1.0、dataset_v1.1。
- 标注规范，说明什么叫通过、部分通过、失败。
- 难度分级：简单、中等、困难、对抗。
- 标签体系：工具调用、参数错误、安全拒答、幻觉、格式错误、成本过高。
- Golden answer，包括标准答案和标准工具轨迹。
- 数据集变更记录，保证不同时间的评测结果可以对比。

目标是：

> 让用例从“测试样本”升级成“可发布、可复现、可对比”的标准评测集。

### 3.3 判分体系还需要更成熟

当前 L1-L6 断言是项目亮点，但真实落地需要更细的评分体系。

需要补充：

- 精确匹配：答案完全一致。
- 语义匹配：答案意思一致但表达不同。
- 工具轨迹匹配：是否调用了正确工具。
- 参数匹配：参数是否正确、是否冗余、是否缺字段。
- 中间过程评分：不只看最终答案，也看工具调用过程。
- 部分得分：工具选对但参数错时，不简单判 0。
- LLM-as-Judge：用大模型辅助判断语义是否等价。
- 人工复核机制：高风险样本不能完全依赖自动判分。
- 置信区间：判断 57.1% / 64.3% 是否具备统计意义。

关键升级方向：

> 从 pass/fail 二元判断升级成多维评分和失败归因。

### 3.4 真实 API 调用稳定性还不够生产级

当前已经能跑真实 API，但真实平台要能大批量、可恢复、可追踪、可控成本地运行。

需要补充：

- API 限流重试。
- 指数退避。
- 超时控制。
- 失败续跑。
- 批量任务并发。
- 请求队列。
- 调用幂等。
- 中断恢复。
- Provider 配额管理。
- Token 成本预算。
- 不同模型返回格式统一。
- 模型版本固定和运行日期记录。

目标是：

> 从“能跑真实 API”升级成“大批量、可恢复、可追踪、可控成本地跑真实 API”。

### 3.5 工具系统还不够接近真实 Agent

当前工具更像本地模拟工具。真实 Agent 评测需要更复杂的工具环境。

需要支持：

- 搜索工具。
- 浏览器工具。
- 文件工具。
- 数据库工具。
- API 工具。
- 代码执行工具。
- 知识库 RAG 工具。
- 企业内部系统工具。
- 权限控制工具。
- 工具失败、慢响应、脏数据、空结果、权限拒绝。

还需要：

- 工具 schema 版本管理。
- 工具调用日志。
- 工具权限边界。
- 工具沙箱。
- 工具结果脱敏。

真实 Agent 测试的核心不是普通问答，而是：

> 模型会不会正确、安全、稳定地使用工具。

### 3.6 安全测试还需要大幅增强

当前已有安全用例，但还可以更接近真实红队评测。

需要补充：

- Prompt 注入。
- 间接 Prompt 注入，例如网页、文档、搜索结果里隐藏攻击指令。
- 越权工具调用。
- 敏感信息泄露。
- 凭证泄露。
- 系统提示词泄露。
- 数据跨用户泄露。
- 恶意文件内容。
- 钓鱼链接识别。
- 拒答边界测试。
- 安全拒答是否过度。
- 工具调用前是否需要用户确认。

信息安全背景可以在这里形成差异化优势。

### 3.7 报告系统还不够产品化

当前能生成报告，但真实平台需要让研发、测试、产品和管理者都能看懂。

需要补充：

- Web Dashboard。
- 模型横向对比。
- 每次运行趋势图。
- 失败用例详情页。
- 工具调用轨迹可视化。
- Token 成本统计。
- 延迟分布。
- 安全风险统计。
- 与上一版本对比：哪些变好了，哪些退化了。
- 一键导出 Markdown / HTML / PDF。
- 给非技术人员看的摘要。
- 给研发看的详细 trace。

报告的目标不是只告诉用户“多少分”，而是告诉用户：

> 为什么失败、失败在哪、该谁修、优先级是什么。

### 3.8 CI/CD 还需要升级成评测门禁

当前已经有 GitHub Actions，这是基础。真实落地要升级成模型上线前质量门禁。

需要补充：

- 每次 PR 自动跑核心用例。
- 每晚定时跑全量用例。
- 低于基线自动失败。
- 延迟或成本超阈值自动报警。
- 新模型接入自动生成对比报告。
- 报告作为 artifact 保存。
- 历史结果可追踪。
- 支持按标签运行，例如 safety、tool_call、regression。

目标是：

> 从“能在 CI 里跑测试”升级成“模型上线前的质量门禁”。

### 3.9 缺少后端服务和数据库

当前项目更像命令行工具。真实平台通常需要服务化。

需要补充：

- FastAPI / Flask 后端。
- 任务提交 API。
- 运行状态查询 API。
- 结果查询 API。
- SQLite / PostgreSQL 存储结果。
- 对象存储保存 trace、日志、报告。
- 用户登录。
- 项目空间。
- 权限管理。
- 多人协作。

目标是：

> 用户上传用例，选择模型，点击运行，系统后台执行并保存结果。

### 3.10 缺少可视化前端

如果要让它像真正的平台，需要一个最小可用前端。

可以做：

- 用例管理页面。
- 模型管理页面。
- 运行任务页面。
- 报告页面。
- 失败样本详情页。
- 工具调用时间线。
- 模型对比表。
- 安全风险看板。
- 成本和延迟图表。

对当前求职目标来说，前端优先级低于评测逻辑、报告和稳定性，但一个最小 Dashboard 会显著增强项目完整度。

### 3.11 缺少可复现部署方式

真实项目要让别人能快速跑起来。

需要补充：

- Dockerfile。
- docker-compose。
- 完整的 .env.example。
- 一键初始化命令。
- 示例 API Key 配置说明。
- 示例 benchmark。
- 本地 mock 模式。
- 真实 API 模式。
- Windows / Linux 兼容说明。

目标是：

> 面试官或同事 clone 项目后，10 分钟内能跑出一份报告。

### 3.12 缺少观测能力

生产系统必须能排查问题。

需要补充：

- 结构化日志。
- request_id / run_id / case_id。
- 每次模型请求的 trace。
- 每个工具调用的输入输出。
- 延迟分解：模型耗时、工具耗时、判分耗时。
- 错误类型统计。
- OpenTelemetry 可选接入。
- 异常报警。

目标是：

> 失败时能判断是模型问题、工具问题、参数问题、断言问题，还是 API 限流问题。

### 3.13 缺少成本控制

Agent 评测会产生真实 API 成本，真实落地必须管成本。

需要补充：

- 每次运行预计 Token。
- 每次运行实际 Token。
- 每个模型单价配置。
- 单次任务成本。
- 总成本。
- 超预算停止。
- 相同请求缓存。
- 小样本快速回归。
- 全量夜间回归。

这个方向很适合面试展示，因为它说明项目不只关注功能，也关注工程成本。

### 3.14 缺少模型注册与适配层

当前支持 DeepSeek / MiniMax，但真实平台需要更通用的模型注册体系。

需要补充：

- Model Registry。
- 每个模型的 provider、base_url、model_name、上下文长度、价格、能力标签。
- 是否支持 tool calling。
- 是否支持 JSON mode。
- 是否支持 streaming。
- 是否支持 vision。
- 统一响应格式。
- provider 特殊错误码处理。
- 模型版本变更记录。

目标是：

> 从“接入两个模型”升级成“可扩展的多模型评测平台”。

### 3.15 缺少人工标注和复核流程

真实评测不能完全自动化，尤其是语义等价和边界场景。

需要补充：

- 人工复核失败样本。
- 人工确认语义等价。
- 标注员意见记录。
- 多人标注一致性。
- 争议样本处理。
- 标准答案修订流程。
- 错误归因审核。

目标是：

> 让自动评测结果可以被人工校准，并持续改进数据集质量。

### 3.16 缺少和真实 Agent 框架的集成

真实公司可能已经有自己的 Agent 框架，评测平台需要能测外部系统。

可以支持：

- LangChain Agent。
- LlamaIndex Agent。
- AutoGen。
- OpenAI Agents SDK。
- 自定义 HTTP Agent。
- MCP 工具。
- 浏览器 Agent。
- 内部 Agent 服务。

目标是：

> 从“我能测自己写的 Agent”升级成“我能测别人的 Agent 系统”。

### 3.17 缺少基线和回归体系

真实落地非常关注新版本有没有退化。

需要补充：

- baseline 结果保存。
- 当前结果与 baseline 对比。
- 通过率下降报警。
- 某类用例退化报警。
- 延迟升高报警。
- 成本升高报警。
- 安全能力退化报警。
- 自动生成 regression report。

目标是：

> 判断模型、Prompt、工具或代码升级后，整体能力是否退化。

### 3.18 缺少更强的失败归因

当前已有三类失败归因，后续可以继续细化。

可以扩展为：

- 模型未调用工具。
- 调用了错误工具。
- 工具顺序错误。
- 参数缺失。
- 参数类型错误。
- 参数值错误。
- 忽略工具结果。
- 最终答案幻觉。
- 安全拒答失败。
- 过度拒答。
- 超出轮次。
- 格式不符合要求。
- Token 超预算。
- 延迟超预算。
- Provider 错误。
- 断言规则过严。

目标是：

> 让失败结果不只停留在 fail，而是能指导后续优化。

### 3.19 缺少权限与合规设计

如果接企业数据，必须考虑权限、安全和合规。

需要补充：

- API Key 加密存储。
- 不把密钥写入日志。
- 用例数据脱敏。
- 模型请求脱敏。
- 用户权限隔离。
- 操作审计。
- 数据保留周期。
- 删除策略。
- 供应商数据出境风险说明。

这部分可以结合信息安全专业背景，形成项目特色。

### 3.20 缺少清晰产品定位

项目最终需要明确服务对象。

可能定位包括：

- 给 AI 应用团队做 Agent 上线前测试。
- 给模型评测团队做多模型横向比较。
- 给安全团队做 Agent 红队测试。
- 给测试开发团队做自动化回归。
- 给算法团队做模型版本回归分析。

当前最适合的定位是：

> 面向 AI Agent 应用上线前的自动化评测与回归测试平台。

## 4. 优先级路线

如果按面试价值和真实落地价值排序，建议按以下顺序推进。

### P0：必须优先做

1. 强化报告系统：HTML 报告、失败归因、模型对比、trace 展示。
2. 强化用例体系：场景分类、难度分级、标签体系、标准答案说明。
3. 稳定真实 API 运行：重试、限流、续跑、成本统计、错误分类。
4. 强化安全测试：Prompt 注入、越权工具、敏感信息泄露。

### P1：明显提升项目完整度

1. 做最小 Web Dashboard：能看运行结果、失败详情和模型对比即可。
2. 支持 baseline 回归：保存历史结果，自动对比退化。
3. 增加模型注册表：统一管理 DeepSeek、MiniMax 和后续模型。
4. 增加 Docker 和一键启动：让别人能快速复现。

### P2：更接近真实生产平台

1. 服务化后端：任务提交、状态查询、结果查询。
2. 数据库存储：保存用例、运行记录、trace、报告。
3. 人工复核流程：处理语义等价和争议样本。
4. 接入真实 Agent 框架或 HTTP Agent：证明能测外部系统。

### P3：长期增强

1. 支持更多工具环境：浏览器、数据库、文件、RAG、企业 API。
2. 引入 LLM-as-Judge。
3. 增加成本预算与缓存。
4. 增加权限、审计、脱敏和合规能力。

## 5. 面试表达

### 5.1 项目当前阶段怎么说

可以这样说：

> AgentEvalLab 当前不是一个简单 Demo，而是一个面向 AI Agent 应用的自动化评测与回归测试原型。它已经支持 YAML 用例管理、真实模型 API 调用、工具调用检查、L1-L6 断言、故障注入、失败归因、模型对比和 CI 自动化测试。

### 5.2 为什么它还不是生产平台

可以这样说：

> 它目前还没有完全达到生产平台水平，主要差距在真实业务场景、数据集标准化、报告产品化、任务服务化、权限合规和大规模稳定运行。但这些差距是我后续规划的方向，不是项目逻辑上的空白。

### 5.3 下一步最想做什么

可以这样说：

> 我下一步会优先强化报告、用例体系和真实 API 稳定性。因为对 Agent 测试来说，最重要的是能稳定复现问题、解释失败原因，并且能对不同模型和不同版本做横向对比。

### 5.4 一句话总结

> 这个项目已经不是玩具 Demo，但还不是完整生产平台。它正处在“能用于面试展示的工程化原型”阶段。后续要围绕真实场景、稳定运行、可解释报告和安全治理继续补强。
