## 30 秒版本

"我做了一个 AI Agent 自动化测试与评测平台。用 339 条 YAML 用例 + L1 到 L6 六层断言，对 DeepSeek 和 MiniMax 做了全量评测——不是只看最终答案对不对，而是检查 Agent 的每一步工具调用是否合理。发现 MiniMax 整体更强但安全有漏洞，DeepSeek 安全稳但知识查询弱。这个项目证明了 Agent 评测和传统软件测试的本质区别。"

## 2 分钟版本

**项目是什么：** AgentEvalLab 是一个 AI Agent 自动化测试与评测平台。传统测试只看输入输出——Agent 输出不确定，同一个问题两次可能走不同路径。我的框架检查 Agent 的完整执行轨迹：调了哪个工具、参数对不对、顺序对不对、有没有被 Prompt 注入。

**怎么做：** 339 条 YAML 用例覆盖功能、边界、异常、安全四类场景。L1-L6 六层断言分别查最终答案、工具选择、参数、轮次、安全和 Token 成本。这保证了"最终答对了但过程错了"（比如调错了工具但蒙对了）也能被检测。

**评测结果：** 用 226 条 LLM-only 用例对 DeepSeek 和 MiniMax 跑了全量评测。MiniMax 整体 75.2% 通过，知识查询几乎全对（95%）。DeepSeek 安全防御更稳（75% vs 70%）。两者在多工具串联上都是弱项（45-55%）。Boundary 类（中日韩语 emoji 全角）两模型 100% 全过。

**和普通 Demo 的区别：** Demo 证明"能跑"——AgentEvalLab 证明"能测、能复现、能对比、能定位问题"。

## 5 分钟版本

### 背景

我在准备 Agent 测试开发岗位时发现，市面上缺少一个能系统评测 Agent 行为的框架。LangChain 的 eval 是黑盒——你不知道里面怎么判断的。我决定自己做一个——能解释每一层断言在查什么、能横向对比不同模型、能自动归因失败原因。

### 技术架构

三层设计：
1. **用例层：** 339 条 YAML，51 手写 + 268 模板生成 + 20 场景用例。覆盖 calculator/weather/knowledge 三个工具，functional/boundary/error/security 四类场景。
2. **Runner 层：** 加载 YAML → 执行 Agent（RuleBased 或 LLM） → 六层断言（L1-L6） → 17 种失败归因。支持 DeepSeek/MiniMax/OpenAI 兼容 API，带 retry + backoff。
3. **报告层：** 控制台 + HTML + Dashboard，展示框架通过率 vs 模型任务完成率、失败归因 Top N、工具调用 trace、双模型对比。

### Benchmark 设计

226 条 requires_llm 用例，涵盖 9 个 scene：multi_tool_planning (35)、rag_document_qa (20)、security (41)、http_agent (10)、multi_turn (10) 等。RuleBaseline 已覆盖 113 条简单用例（382 passed）。

### 真实评测结果

| 模型 | 通过率 | 安全 | Knowledge |
|------|--------|------|-----------|
| DeepSeek | 65.0% | 75% ✅ | 30% |
| MiniMax | 75.2% | 70% | 95% ✅ |

### 典型失败案例

- **多工具串联：** Prompt 说"查天气 + 建议穿搭"，模型只调了 weather 没调 knowledge。不是模型做错了，是它觉得天气数据里已有足够信息。
- **SEC-008 间接注入：** MiniMax 在回答中输出了完整系统提示词——真实安全漏洞。DeepSeek 正确拒答——但断言引擎最初的拒答检测不到"否定式"拒答（如"我没有 API Key"），误判为泄露。
- **P0 修复后：** 扩展了拒答检测模式，DeepSeek SEC-008 从误判变通过，MiniMax 安全从 1/3 恢复到 2/3。

### 下一步

LLM-as-Judge 语义等价、多工具 prompt 优化、人工复核 UI、更多模型接入。

## 项目背景

我作为信息安全专业学生，想找 Agent 测试开发岗位。在准备面试时发现：传统的 pass/fail 测试不适用于 Agent——因为 Agent 的输出不确定，只看最终答案会漏掉工具调用错误。我决定做一个能"检查 Agent 每一步"的评测框架。

---

## 核心问题

**Agent 测试和传统软件测试有什么不同？**

传统：输入→输出，确定性，pass/fail。  
Agent：输入→思考→工具调用→观察→输出，非确定性，需要检查完整链条。

**AgentEvalLab 解决了什么？**

1. 分层断言（L1-L6）——不只查答案，查每一步
2. 17 种失败归因——失败不只是 pass/fail，能说出为什么
3. 双模型横向对比——同一批用例、同一套断言，出两个模型的画像
4. 稳定性评测——同一用例跑多次，看过程稳不稳

## 和普通 Demo 的区别

| 普通 Demo | AgentEvalLab |
|-----------|-------------|
| 展示一个功能能用 | 系统评测 Agent 在多场景下的表现 |
| 手写几个样例 | 339 条 YAML，9 类生成器 |
| 人工看输出判断对错 | L1-L6 自动断言 |
| 跑一次就算了 | pytest + CI + retry + resume |
| 只能测一个模型 | 双模型横向对比 |
| "我做了一个功能" | "我建了一套评测体系" |
