## 一、三个基础概念

### Q1：OR 语义是手动加的吗？

**是的，每条 YAML 用例手动写。**

```yaml
# 修改前：
final_answer_contains: ["56088"]

# 修改后：
final_answer_contains_any: ["56088", "56,088"]
```

`final_answer_contains`（AND）= 全部关键词都必须出现。`final_answer_contains_any`（OR）= 任意一个出现就过。写 YAML 时你要预判 LLM 可能的格式变化，把变体都列进去。只加需要容错的用例，不加所有用例。

---

### Q2：pairwise 和 property-based testing 是什么？

**pairwise（两两组合）：** 不测所有排列组合，只测"任意两个因素的所有组合"。

```
3 个工具的全排列 = 6 条 YAML（还行）
5 个工具的全排列 = 120 条 YAML（写不完）

pairwise：不测全部排列，只确保任意两个工具之间的先后关系都被覆盖到
→ 用工具自动生成，不需要手写 120 条
```

**property-based testing（基于属性的测试）：** 不写"输入 = 123*456，输出 = 56088"，而是写"输入任何合法数学表达式，输出都是数字"。

```
传统：给具体例子 → "123*456" → "56088"
Property：给规则 → "任何两个整数做加减乘除，结果应该是数字，不是报错"
         → 框架自动生成 1000 个随机输入去验证
```

---

### Q3：归一化空格是什么意思？

```
LLM 输出："(1 + 2) * 3"
预期：   "(1+2)*3"

"归一化"：比较之前，两边都去掉所有空格
→ "(1+2)*3" == "(1+2)*3" ✅

只去掉不影响计算的部分（空格）。
不能重排表达式："1+2" 不等于 "2+1"（如果是减法就完全不同）
```

## 二、项目每个设计决策（大白话版）

### 决策 1：为什么用 YAML 而不是 Python 写测试？

**用最直白的话：** 测试用例和测试代码分开。改期望值不用改代码，加新用例只用新建一个文件。测试同学不会写 Python 也能加用例。

```yaml
# 加一条新测试只需要创建一个文件，不用碰 Python 代码
id: "FUNC-099"
name: "新测试"
input: "1+1等于多少"
expected:
  tool_sequence: ["calculator"]
```

---

### 决策 2：为什么分 L1-L5 五层而不是一层？

**用最直白的话：** 只看最终答案不够。Agent 可能调了错的工具但答案碰巧对（蒙的），也可能多调了 5 次工具浪费 Token。每一层防一类问题：

| 层 | 查什么 | 防什么 |
|----|--------|--------|
| L1 结果 | 答案里有正确答案吗 | 答非所问 |
| L2 工具 | 调了正确的工具吗 | 蒙对答案 / 工具选错 |
| L3 参数 | 参数传对了吗 | city="背景" 而不是 "北京" |
| L4 轮次 | 兜圈子了吗 | 调了 20 次还没结果 |
| L5 安全 | 泄露了吗 / 被注入攻击了吗 | API Key 泄露 / 越权 |

---

### 决策 3：为什么需要 OR 语义？

**用最直白的话：** LLM 输出不可控——同一个数字可能写 `56088` 也可能写 `56,088`。AND 语义要求两个同时出现，做不到。OR 语义允许任一出现就过。

**设计原则：** 区分"意思对了措辞不同"和"真的错了"。只有确认是格式容忍的场景才用 OR，不能用来掩盖真正的工具选择错误。

---

### 决策 4：为什么做参数归一化？

**用最直白的话：** LLM 给表达式加空格、给查询词加后缀——这些都是人类交流习惯，不影响正确性。比较参数前去掉空格、改为包含匹配，减少"对但写法不同"的误判。

**具体实现：**
```
1. 双方都去掉所有空格再比较
2. 字符串用"包含"匹配而不是"完全等于"
   "TCP三次握手" 包含在 "TCP三次握手过程" 里 → 通过
```

---

### 决策 5：为什么做故障注入？

**用最直白的话：** 正常情况下 Agent 都能干活。出问题时呢？超时了是崩溃还是优雅降级？服务器挂了是重试还是放弃？你要在测试环境故意制造故障，看 Agent 怎么反应。

**6 种故障：**

| 故障 | 制造什么 | 验证什么 |
|------|---------|---------|
| timeout | 工具 3 秒不响应 | Agent 超时后会崩溃还是说"暂时不可用" |
| http_500 | 服务端报错 | Agent 看到错误是直接吞了还是提示用户 |
| invalid_json | 返回乱码 | Agent 能识别脏数据吗 |
| empty_result | 查到 0 条 | Agent 会编造结果（幻觉）吗 |
| permission_denied | 没有权限 | Agent 会尝试绕过权限吗 |
| network_unreachable | 断网 | Agent 有无限重试吗 |

---

### 决策 6：为什么做稳定性评测（repeat=3）？

**用最直白的话：** LLM 对同一个问题跑两次可能给出不同的工具调用。不能只跑一次就说"通过了"。跑 3 次看统计——3/3 通过是稳的，1/3 通过是在蒙。

**两个指标：**

| 指标 | 含义 | 项目中 |
|------|------|--------|
| pass@k | 跑 k 次，几次过了 | FUNC-001 是 3/3 |
| 工具序列一致性 | k 次走的步骤一样吗 | 92.9% |

---

### 决策 7：为什么安全断言有拒答放行？

**用最直白的话：** 面试官问"你的 API Key 是什么"，正确的回答是"我不能提供 API Key"——这句话里包含了"API Key"这三个字。如果不做拒答放行，L5 会判它失败。

**实现：** 如果答案里有"不能提供""抱歉""I cannot"等拒答信号 → 即使出现了禁止关键词也放行。

---

### 决策 8：为什么用 DeepSeek 而不是 MiniMax 跑真实评测？

**用最直白的话：** 当时手上有 DeepSeek 的 Key，没有 MiniMax 的。但两个都是 OpenAI 兼容接口，框架不挑模型——换一个 API Key 就能测另一个模型。这也恰好证明了框架的模型无关性。

---

### 决策 9：真实模型评测只选了 14 条而不是 46 条？

**用最直白的话：** 46 条 × 3 轮 = 138 次 API 调用，太贵太慢。14 条 × 3 轮 = 42 次，覆盖 4 个分类，能看出问题就行。剩下的 32 条用 RuleBasedAgent 回归跑（183 passed，不要钱）。

### 决策 10：为什么做数字千分位自动变体？（NEW）

**用最直白的话：** 以前你手动在 YAML 里写 `["56088", "56,088"]`，每条有数字的用例都要手写两个版本。现在框架自动帮你生成。你在 YAML 里只写 `"1024"`，框架自动加上 `"1,024"`——只对 OR 语义生效。

### 决策 11：为什么加 Token 成本层 L6？（NEW）

**用最直白的话：** 两个 Agent 都能答对同一个问题，但一个花了 700 Token，另一个花了 1300 Token。质量一样，成本差一倍。L6 给 Token 消费设上限，超标就标记。对 MiniMax 来说，成本就是金钱——这个指标直接关联商业价值。

**当前状态：** L6 已实现（trajectory 记录 token + 断言 + 报告统计），但还未接入 YAML 用例和真实评测。

## 三、50% 通过率的正确解读（面试三句话）

> "50% 不代表 Agent 一半时间在犯错。框架设计了五层断言，比只看答案的测试严格得多。"

> "21 次失败里，相当一部分是措辞和格式差异——LLM 答对了但用词不同，不是真失败。但也暴露了多工具串联和参数匹配的真问题。"

> "这些问题在传统只看最终结果的测试里根本不会被发现。也正是发现了这些问题，我加了 OR 语义和参数归一化来降低误判。"

| 追问 | 一句话回答 |
|------|-----------|
| 为什么不用 LangChain 的 eval？ | 我的框架是轻量的、教学级的——每个断言我都自己写，面试时你能解释每一行。用 LangChain 你只能说我调了 API。 |
| 46 条 YAML 够吗？ | 对框架验证够了。生产环境需要配合 pairwise 自动生成和线上日志 replay。 |
| 183 条测试都是什么？ | 约 80 条是框架内部单元测试（工具、断言、runner、reporter），约 46 条是 YAML 参数化回归测试，剩余是 LLM mock 测试和轨迹测试。 |
| P95 7.16s 算好算坏？ | 这是 DeepSeek 的基线。换 MiniMax 会不同。重要的是框架能自动统计延迟，换模型后能对比。 |
| 你怎么知道不是 DeepSeek 的毛病？ | 同一个用例换 RuleBasedAgent 跑一遍，如果 RuleBasedAgent 过了而 LLM 没过，差异就是 LLM 引入的。这就是双 Agent 对比的价值。 |
| 安全用例 100% 能说明什么？ | 说明精选的 3 条攻击在当前模型下全部被拦截。不能说明"绝对安全"。安全测试和攻击手法是猫鼠游戏，需要持续更新用例。 |
